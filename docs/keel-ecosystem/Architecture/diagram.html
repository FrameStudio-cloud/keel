Keel Ecosystem — Architecture Diagram

One Supabase project · Two app types · N shops · End-to-end data flow

Supabase (Single Project — hmcowpwfefeeossztuem)
Database
20+ Tables
shops, products, catalogue, banners, sales, expenses, store_settings, categories, category_attributes, product_attribute_values, stock_movements, page_views, announcements, users, posts, payments
Auth
Supabase Auth
Email/password login, Google OAuth, session management. Accessed via direct REST API (not GoTrueClient). RLS disabled (all queries use shop_id filtering instead)
RPCs
Postgres Functions
get_dashboard_summary, get_profit_margins — server-side aggregations called by the dashboard
↑ All reads/writes go to the same Supabase project, filtered by shop_id

Context
AuthContext + SettingsProvider
Auth state (login/logout via REST API), settings (fetched once, consumed via useSettings), theme toggle (light default), currency, payment config
Modals
8 Modal Components
AddProduct, EditProduct (with "Other" attribute option), LogSale, BarcodeScanner, ImageUploader, DeleteShop, SignOut — all with focus traps

12 Settings Components
7 tabs (Store, Preferences, Notifications, Billing, Security, Data, Danger Zone) + shared SectionCard, TabButton, SaveBar, styles
↑ writes product data, sales, etc.

Automation Gap (Not Yet Built)
The Manual Step
Today, publishing a product writes to the catalogue table in Supabase — but the public site won't show it until someone manually runs create-mini-catalogue and redeploys to Vercel.
To automate: Supabase webhook → Vercel API → auto-deploy → *.keel.framestudio.co.ke   ·   Or: mini-catalogue reads live from Supabase (no deploy needed)
